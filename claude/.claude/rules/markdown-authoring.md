# Markdown トンマナガイド

## 概要

`.md` ファイルを生成・編集する際の文体・語彙・記法ルール。AI 生成にありがちな硬い表現や過剰な装飾を抑制し、読みやすい日本語を維持する。

## 認知負荷の低減

読者が最小限の労力で内容を把握できることを最優先にする。以下のルールはすべてこの原則に基づく。

- そのドキュメントで一番重要な情報を先頭に置く。読者が最初に目にする場所に核心がくるように
- 文章で説明できても、表や箇条書きの方が早く読めるならそちらを使う
- 1 セル・1 箇条書きに載せる情報は 1 つ。複数の性質が混ざっているならカラムや行を分けて整理する
- 表のカラム名は読者の問いに対応させる。何を知りたくてこの列を見るのかが名前から伝わるように
- 一文は短く。接続が多い長文より、短文 + 箇条書き
- 詳細・補足・参考情報は本文の後半や付録に回し、冒頭の見通しを妨げない

### 強調・装飾

- 太字（`**...**`）は基本的に使わない。地の文で十分伝わるなら強調不要
- 太字見出し（`**...**:`）で段落を区切りすぎない。地の文で流せるなら流す
- `—`（em dash, U+2014）はどの用途でも使わない。用語と説明を繋ぐインライン記法、引用の出典表記、区切り文字のいずれも禁止。説明が必要なら箇条書きのネストや補足ブロックで書く
- `§`（セクション記号）は使わない。セクションへの参照はアンカーリンク `[ドキュメント名](path.md#anchor)` で書く
- 絵文字を装飾目的で使わない。ステータス表現も絵文字ではなくテキストで書く

### 改行・余白

- 明示的な改行には `<br>` タグを使う。末尾スペース 2 つによる改行は使わない
    - 末尾スペースは不可視で、エディタの trailing whitespace 除去や Claude の Edit ツールで消されやすい
- 情報密度の高いブロックの直後は `<br>` や空行で視覚的な余白を意識する
- 長い段落は避ける。話題や論点の区切りで改行・空行を入れる
    - 目安: 3〜4 文程度でひとまとまり。それ以上続くなら段落を分けるか箇条書きに切り替える

## 文体

- 常体（だ/である）ベースだが、硬い文語表現は避ける
    - OK: 「〜になる」「〜できない」「〜は未定」「〜で補完」
    - NG: 「〜となる」「〜である」「〜と言える」「〜が求められる」
- 体言止めを活用: 「〜ブラックボックス」「〜未定」
- 文末を「:」（コロン）で終わらせない

## 語彙

- 日常寄りの語を選ぶ
    - 「規定する」→「決める」、「主題」→「テーマ」、「格段に」→ 削除
    - 「本ドキュメント」→「このドキュメント」
- 状態表現はそのまま使う: 「〜が有力」「〜は未定」「〜で補完」

## 和欧混植

日本語サイエンスライティングの和欧混植ルールに準拠する。ここに明示されていないケースもサイエンスライティングの慣例に従う。

### スペーシング

日本語と半角英数が隣接する場合は半角英数の前後に半角スペースを入れる。

- OK: `task skill と reference skill の 2 種類がある`
- NG: `task skillとreference skillの2種類がある`

例外（スペースを入れない）:

- 句読点の直前: `これがその skill。`
- インラインコード前後: `` `claude -p` で実行する``（バッククォート自体が視覚的区切りになる）

### 括弧

- 括弧内が半角英数 → 半角括弧。外側にスペース、内側にスペースなし: `セッション (Cookie) とは無関係`
- 括弧内が日本語等の全角文字 → 全角括弧: `Auth0 セッション（最大 7 日）より`

## インデント

- 箇条書きのネストは 4 スペースインデント。2 スペースだと renderer によってはネストと解釈されない
    - OK: 4 スペース
    - NG: 2 スペース

## 引用ブロック・参照

引用ブロック (`>`) の用途を以下に限定する。それ以外の目的で引用ブロックを使わない。

### 補足ブロック

本文の理解を助ける補足説明に使う。何も引用せず、見やすさのために引用ブロックを流用する。補足内にリンクがあっても、それが補足内容の裏付けであり本文のエビデンスでなければ補足のまま。

- 1 行目: `> 補足: タイトル<br>`（`<br>` タグで改行）
- 2 行目: `> 本文...`
- タイトルに `**`（太字）は使わない
- 1 つの補足ブロックは 2 行（タイトル + 本文）で構成する
- 1 つの補足ブロックで複数のトピックを説明しない。補足したいことが複数あればブロックを分ける

```markdown
> 補足: Confidential Client<br>
> OAuth 2.0 では...（本文）
```

### 引用

外部ドキュメントの文言をそのまま引用する。補足説明は加えず、文言の転記と出典のみ。ラベルやタイトルは不要。

- 引用文は `"..."` で囲む
- 出典は `出典: [タイトル](URL)` の形式で最終行に置く
- 補足説明を加えたい場合は情報源ブロックを使う

```markdown
> "Each execution of a trigger must complete in 20 seconds or less."
>
> 出典: [Actions Limitations](https://auth0.com/docs/customize/actions/limitations)
```

### 情報源ブロック

本文の主張や判断を支えるエビデンスを記録する。補足説明を加えながら情報源を示す。文言の引用を含めてもよい。

- 1 行目: `> 情報源: タイトル<br>`（`<br>` タグで改行）
- 2 行目以降: 補足説明、根拠、リンク等
- リンク先のどこを根拠にしているかを本文中に書く

```markdown
> 情報源: Auth0 Actions の制約<br>
> Actions の実行時間は 20 秒以内に制限されている。user_metadata の更新にはレート制限もあるため、高頻度の書き込みには外部 DB を使う必要がある。（[Actions Limitations](https://auth0.com/docs/customize/actions/limitations)）
```

### GFM Alerts

注記・注意書き・アーカイブ注記など、読者への通知に使う。GitHub でネイティブレンダリングされる。

- 使えるタイプ: `NOTE`, `TIP`, `IMPORTANT`, `WARNING`, `CAUTION`
- 主な用途: ドキュメントのステータス（移行済み、WIP 等）、セクションのスコープ注記、注意事項

```markdown
> [!NOTE]
> この資料は [マイルストーン全体像](../planning/epic-overview.md) に移行済み。
> 移行計画 (Phase X) など、新エピック資料に含まれていない情報の参照元として残している。
```

```markdown
> [!IMPORTANT]
> 設計タスクのみ。実装・検証タスクはフェーズ 2.3 に分離。
```

### ※ 参照

関連ドキュメントへの参照先を示す。インラインリンクで地の文に組み込むのが基本だが、うまく書けない場合に本文ブロックの直後に置く。引用ブロックは使わない。

- `※ 説明文（[参照先タイトル](URL)）` の形式
- 複数ある場合は `※` を連続して書き、各行の末尾に `<br>` タグで改行する

```markdown
※ 詳細は [ゲストセッション・チェックアウト設計](./guest-session-design.md) を参照<br>
※ App Proxy の仕様は [Shopify プラットフォーム制約・既知問題](../investigation/shopify-platform-constraints.md) Section 1 を参照
```

### 配置

引用ブロック（補足・情報源・引用）は、それが支える対象の記述の**後**に置く。主張や数値を先に書き、その根拠を後から示す。

### 使い分け

判断基準: 本文に対して **何を提供しているか** で分ける。

| 本文との関係 | 記法 | 例 |
|---|---|---|
| 理解を助ける説明 | `> 補足:` | 用語解説、仕組みの背景 |
| 主張を支えるエビデンス | `> 情報源:` | 設計判断の根拠、外部制約 |
| 外部文言のそのままの転記 | `> "..."` + `> 出典: [出典]` | 公式ドキュメントの引用 |
| 読者への通知 | `> [!NOTE]` 等 | ステータス、スコープ注記 |
| 参照先の提示 | `※` | 「詳細は〜を参照」 |

## エビデンス

判断や選択の根拠、具体的な数値には情報源を付ける。

- 「〜を採用」「〜が有力」「〜が必要」等の判断には、情報源ブロック（`> 情報源:`）で根拠を示す
- 具体的な数値（レート制限、TTL、レイテンシ、SLA、コスト等）には出典リンクを付ける
- 不確実な記述（「おそらく」「雰囲気」等）を残す場合も、判断の手がかりとなる参照リンクを添える

## lint エラーの修正方針

markdownlint のエラーには形式的に直せるものと、意味的な解決が必要なものがある。

形式的修正で済むもの（機械的に置換してよい）:

- em dash → 括弧や箇条書きに置き換え
- `./` プレフィックスの追加

意味的な解決が必要なもの（表面的な書き換えで lint を黙らせてはいけない。以下は代表例であり、都度「このエラーは形式的に直せるか、意味的な判断が必要か」を考える）:

- TODO 残存: TODO の内容を実際に解決するか、未解決事項として管理する。「TODO」の文字列を消して別の言い方にするだけでは解決にならない
- プレーンテキスト .md 参照: 適切なリンクテキストと相対パスを確認してリンク化する
- 壊れたアンカー: 対象ファイルの見出しを確認して正しいアンカーに修正する

## AI パターン抑制

AI 生成にありがちな文体パターンを避ける。検出・修正の詳細は `/md-humanize` スキルを参照。

- 接続詞 (`さらに` / `加えて` / `また` / `そのため`) を 2 文連続で文頭に使わない。接続詞なしで文意が通るならそのまま書く
- 回りくどい繋辞を使わない: `〜として位置づけられている` → `〜だ`、`〜と考えられます` → `〜と考える`、`〜と言えるでしょう` → 削除
- 根拠のない意義の誇張をしない: `非常に重要`, `画期的な`, `極めて`, `注目すべき`（内容自体で示す）
- 定型的な `まとめ` / `おわりに` で本文を繰り返さない。結論が新情報やアクションを加えないなら省略する
- リストを 3 つに揃えない。自然な項目数で書く。3 つ目が曖昧なら削除する

## ファイル間リンク

関連する記述や参照が必要になったら、必ず inline link または引用ブロック系（情報源ブロック、補足ブロック等）でリンクする。ドキュメント間の依存関係は本文中のリンクで表現する。frontmatter 等での別管理はしない。

プレーンテキストでの `.md` ファイル名の言及は禁止。

「関連ドキュメント」テーブル（ADR 等の末尾）は本文中で参照したドキュメントの集約インデックスとして使う。テーブルにしか載っていない参照は禁止。本文のどこかで必ず inline link または引用ブロックで言及した上で、テーブルにも記載する。

- リンク文字列にはファイル名ではなく、対象ドキュメントの `# タイトル`（h1）をそのまま使う
    - h1 がないファイルの場合はファイル名をリンク文字列にする
- パスは相対パスで、同一ディレクトリなら `./`、別ディレクトリなら `../dir/` を使う。`./` プレフィックスは必須
- セクションへの参照はアンカーリンクにする。`[Doc](path.md) Section 3` ではなく `[Doc](path.md#3-セクション名)` と書く
- バッククォート（`` `file.md` ``）での言及はリンクにならないため使わない
- タイトルを変更した場合は、被リンク側のリンク文字列もすべて更新する
    - 被リンク箇所が多い・特定が難しい場合はユーザーに確認してから対応する
- 例: `[API 連携・認証経路 設計](./api-integration-design.md)` を参照
